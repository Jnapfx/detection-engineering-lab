# 1. Architecture

Microsoft Sentinel is a cloud-native SIEM and SOAR platform used to collect, analyze, detect, investigate, and respond to security events.

## Data Flow

Data Source → Data Connector → Azure Monitor → Log Analytics Workspace → Microsoft Sentinel → Alerts → Incidents

## Component Roles

- Data Connector: Responsible for ingesting data from sources  
- Azure Monitor: Processes and routes telemetry data  
- Log Analytics Workspace: Stores logs  
- Microsoft Sentinel: Detects threats and enables response  

## Why It Matters

Understanding this flow is essential for troubleshooting ingestion issues and answering SC-200 exam questions related to architecture.

---

# 2. Data Connectors

Data connectors integrate data sources into Sentinel.

## Types

- Microsoft-native connectors  
  - Azure Activity  
  - Microsoft Defender XDR  
  - Microsoft Entra ID  

- Agent-based connectors  
  - Windows Security Events  
  - Linux Syslog  
  - Azure Monitor Agent (AMA)  

- Third-party connectors  
  - Syslog  
  - CEF  
  - API integrations  

## Key Concept

A connector showing "Connected" does not guarantee data ingestion. Logs must be validated in tables.

## Real SOC Usage

Before building detection rules, analysts confirm that logs are actively being ingested.

# 3. Log Ingestion

Log ingestion is the process of collecting and storing security data.

## Process

1. Event generated by system  
2. Connector collects data  
3. Azure Monitor processes it  
4. Stored in Log Analytics Workspace  
5. Used by Sentinel  

## Example

A failed login (Event ID 4625) is generated on a VM and ingested into Sentinel, where it becomes available for queries and alerts.

# 4. Azure Monitor Agent vs Legacy Agent

## Azure Monitor Agent (AMA)

- Recommended  
- Modern data collection  
- Better scalability  

## Legacy Agent

- Deprecated  
- Still present in older environments  

## Exam Note

Azure Monitor Agent is the correct answer for modern log collection.

# 7. KQL Fundamentals

KQL is used for querying log data.

## where

```kql
SecurityEvent
| where EventID == 4625
```

## summarize

```kql
SecurityEvent
| summarize Count = count() by EventID
```

## project

```kql
SecurityEvent
| project TimeGenerated, Computer, EventID
```

## join

```kql
TableA
| join kind=inner TableB on Account
```

KQL is used for detection, investigation, and threat hunting.

# 12. SOC Use Case: Brute Force Attack

## Scenario

A VM is exposed via RDP (3389) and receives multiple login attempts.

## Detection Strategy

- Monitor Event ID 4625  
- Group by IP  
- Detect high frequency  

## Query

```kql
SecurityEvent
| where EventID == 4625
| summarize Attempts = count() by IPAddress
| sort by Attempts desc
```

## Investigation

1. Identify attacker IP  
2. Review login attempts  
3. Correlate logs  

## Mitigation

- Restrict NSG  
- Limit IP access  
- Add MFA
