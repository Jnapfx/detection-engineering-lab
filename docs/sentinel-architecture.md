# 1. Introduction to Microsoft Sentinel

Microsoft Sentinel is a cloud-native **Security Information and Event Management (SIEM)** and **Security Orchestration, Automation, and Response (SOAR)** platform built on Microsoft Azure.

It allows security teams to:

- Collect security logs
- Detect threats
- Investigate incidents
- Automate responses

Sentinel provides a centralized platform for monitoring security across:

- Cloud environments
- On-premises infrastructure
- Applications
- Identity systems
- Network devices

---

# 2. What is a SIEM?

A **SIEM (Security Information and Event Management)** system collects and analyzes logs from multiple sources to detect suspicious activity.

Core capabilities include:

- Log collection
- Event correlation
- Threat detection
- Alert generation
- Incident investigation

Traditional SIEM systems required on-premises infrastructure, but Microsoft Sentinel is fully cloud-native.

---

# 3. Benefits of Microsoft Sentinel

Microsoft Sentinel offers several advantages compared to traditional SIEM platforms.

Key benefits include:

- Cloud scalability
- No infrastructure management
- Built-in threat intelligence
- Integration with Microsoft security tools
- Advanced analytics using KQL
- Automation with playbooks

Because Sentinel is cloud-based, it can analyze **massive amounts of data quickly**.

# 4. Sentinel Architecture Overview

Microsoft Sentinel operates on top of **Azure Log Analytics Workspace**.

The architecture can be represented as:

```
Data Sources
     ↓
Data Connectors
     ↓
Log Analytics Workspace
     ↓
Analytics Rules
     ↓
Incidents & Alerts
     ↓
Investigation
     ↓
Automation / Response
```

Each component plays a critical role in threat detection.

# 7. Log Analytics Workspace

Log Analytics is the central storage and analysis engine for Sentinel.

It stores all ingested log data.

Security analysts can query logs using **Kusto Query Language (KQL)**.

Example log tables include:

- SecurityEvent
- SigninLogs
- AzureActivity
- OfficeActivity

Log Analytics enables powerful search and analysis capabilities.

---

# 8. Kusto Query Language (KQL)

KQL is used to query and analyze logs in Microsoft Sentinel.

Example query for failed login attempts:

```kql
SigninLogs
| where ResultType != 0
| project TimeGenerated, UserPrincipalName, IPAddress
| limit 10
```

This query retrieves login failures.

Another example detecting multiple failed logins:

```kql
SecurityEvent
| where EventID == 4625
| summarize FailedAttempts = count() by Account
| sort by FailedAttempts desc
```

KQL allows analysts to perform advanced threat hunting.

# 9. Analytics Rules

Analytics rules detect suspicious patterns in logs.

They generate alerts when predefined conditions are met.

Types of analytics rules include:

- Scheduled query rules
- Near real-time rules
- Fusion detection
- Machine learning detections

Example detection scenarios:

- Multiple failed login attempts
- Suspicious geographic logins
- Malware activity
- Privilege escalation attempts

Alerts generated by these rules create incidents for investigation.

---

# 10. Alerts vs Incidents

It is important to understand the difference between alerts and incidents.

## Alert

An alert is generated when a detection rule identifies suspicious activity.

Example:

```
Alert:
Multiple failed login attempts detected
```

Alerts represent potential security issues.

---

## Incident

An incident is a group of related alerts.

Example:

```
Incident:
Credential Brute Force Attempt
 ├── Alert 1: Multiple failed logins
 ├── Alert 2: Suspicious IP address
 └── Alert 3: Unusual login location
```

Grouping alerts into incidents simplifies investigation.

# 12. Threat Hunting

Threat hunting involves proactively searching for hidden threats.

Analysts use **hunting queries** to detect suspicious patterns.

Example hunting query:

```kql
SigninLogs
| summarize LoginAttempts = count() by IPAddress
| where LoginAttempts > 20
```

This query detects potential brute force attacks.

Threat hunting helps identify attacks that bypass automated detections.

# 14. Automation and SOAR

Microsoft Sentinel includes **Security Orchestration, Automation, and Response (SOAR)** capabilities.

Automation is implemented through **playbooks** built using Azure Logic Apps.

Playbooks can automatically perform actions such as:

- Blocking malicious IP addresses
- Disabling compromised accounts
- Sending alerts to security teams
- Creating tickets in incident management systems

Example automation flow:

```
Incident Detected
     ↓
Playbook Triggered
     ↓
User Account Disabled
     ↓
Security Team Notified
```

Automation significantly reduces response time.
